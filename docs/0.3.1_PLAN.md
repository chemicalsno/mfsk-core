# 0.3.1 — uvpacket redesign

## Context

The 0.3.0-cycle uvpacket protocol was abandoned after an honest
airtime comparison showed AFSK 1200 / AX.25 was 5–10× faster on
clean channels for the target signed-QSL payload. The
"drop-in-AFSK" pitch could not survive the comparison.

This release cycle is purely additive (new module, new feature
flag, new `ProtocolId` variant, no changes to existing public
API), so the redesign ships as a 0.3.1 patch release on top of
0.3.0 rather than a 0.4.0 minor bump. Pre-1.0 minor bumps stay
reserved for the next breaking change.

The 0.3.1 cycle rebuilds uvpacket from scratch under a reframed
goal:

- **Primary goal**: experimental verification of modulation
  robustness in NFM-band fading channels.
- **Use cases**: private-group ham messaging on a simplex
  frequency, with two flagship applications — (1) signed QSL
  exchange (Bitcoin/Monacoin "Sign Message" compatible), and
  (2) APRS-style short-message position reporting *within the
  group* (not interoperable with the public APRS network).
- **Adoption is explicitly out of scope**. Both endpoints install
  the same software. No coexistence with public AX.25
  infrastructure (digipeaters, igates) is required.

## Design summary

| Layer | Choice |
|---|---|
| Modulation | 4-GFSK, h=0.5, BT=0.5, R_s = 1200 baud |
| Tone centre / spacing | 1700 Hz centre; tones at 800 / 1400 / 2000 / 2600 Hz |
| Channel rate | 2400 ch bps |
| Audio passband | ~2.0 kHz, fits narrow-FM voice channels |
| Inner FEC | CCSDS 231.1-O-1 short LDPC, base block (n=256, k=128) |
| Modes (rate / net bps) | Robust 1/2 (1200), Standard 2/3 (1600), Fast 3/4 (1800) |
| Higher-rate construction | Puncturing the rate-1/2 mother code |
| Sync | Costas 7×7 array (lifted from FT4 / FT8), 7 symbols per Costas |
| Sync placement | Frame head + frame tail + mid-frame every 4 LDPC blocks |
| Interleaver | Block-interleaver across all LDPC blocks in a frame |
| Frame length | Variable, 1 to 16 LDPC blocks (payload 16 to 256 bytes) |
| Frame header | Mode (2b) + block count (4b) + app type (4b) + sequence (6b) + CRC-16 (16b) = 4 bytes |
| Application API | **Byte pipe**: `encode_frame(&[u8]) → Vec<f32>` and `decode_frame(&[f32]) → Vec<(app_type, Vec<u8>)>`. No `MessageCodec` indirection — applications choose their own serialisation (CBOR, JSON, Protobuf, raw). |
| Receiver | Non-coherent 4-tone energy detection; no equalizer |
| ARQ | Time-bounded LDPC iteration; timeout → request retransmit |
| PTT | IC-705 BLE control (webft8 pattern) + fixed delay before audio |
| Target channels | AWGN; 1 Hz Rayleigh primary, 0.5–20 Hz sweep for characterisation |

### Why no equaliser

The audio domain after FM detection has no preserved RF phase, so
the typical linear / DFE / MMSE equalisers do not apply. The three
remaining acquisition tasks — frequency offset, symbol timing, and
slow AGC — are handled by the head, tail, and mid-frame Costas
arrays. Fade bursts (1 Hz Rayleigh, ≤ 30 % of frame at burst peak)
are recovered by the block interleaver + rate-1/2 LDPC, not by
channel inversion.

### Why no `MessageCodec` indirection

The existing `MessageCodec` trait is shaped around the WSJT-77
77-bit fixed-length callsign-packed message that FT8/FT4/FST4 use.
It does not fit a multi-block, variable-length, application-agnostic
byte payload. The new uvpacket bypasses this layer entirely and
exposes a byte-pipe API. Applications (signed QSL, position
beacons, free text, ACKs) choose their own serialisation and
identify themselves via the 4-bit `app type` field in the frame
header:

| `app_type` | Reserved for |
|---:|---|
| 0 | Raw / tagless — application uses its own magic bytes |
| 1 | Signed QSL (Bitcoin/Monacoin "Sign Message" + JSON) |
| 2 | Position report (compressed lat/lon + altitude + status) |
| 3 | Text message |
| 4 | ARQ ACK / NAK |
| 5–15 | User-defined |

`app_type = 0` preserves the option of pure tagless operation —
applications that want to manage their own dispatching can do so
without involving the modem layer.

The `PacketBytesMessage` codec (added in 0.2.1 for the abandoned
old uvpacket) is removed in this cycle. It has no users outside
the reverted old uvpacket.

### Why CCSDS LDPC

A new generator matrix is unnecessary. CCSDS 231.1-O-1 publishes
short-block LDPC codes (128, 256, 512 information bits) with H
matrices designed and characterised for deep-space telemetry —
the same regime (short messages, severe fading, low SNR floor) we
need for marginal NFM channels. The published H is dropped in
verbatim. mfsk-core already has LDPC family infrastructure
(`Ldpc174_91`, `Ldpc240_101` under `LdpcParams`) so adding one
more concrete params type is small.

### Why 4-GFSK at 1200 baud

- **Audio bandwidth ≈ 2 kHz fits narrow-FM voice** (cheap HT
  passband 300 Hz – 2.7 kHz). The 1700 Hz centre keeps the lowest
  tone (800 Hz) clear of HT high-pass at 300–500 Hz.
- **2 bits / symbol** at 1200 baud → 2400 ch bps. With rate-1/2
  LDPC this yields **1200 net bps = AX.25 throughput parity**.
- **Non-coherent detection** is robust to FM-discriminator phase
  destruction; no carrier recovery needed.
- **h = 0.5** gives orthogonal tones (energy-detector friendly)
  with modest BW.

### Throughput vs AX.25 / AFSK 1200

For a 214-byte signed-QSL payload:

| Transport | Net bps | Frame airtime | Notes |
|---|---:|---:|---|
| AX.25 / AFSK 1200 | ~1100 | ~2.0 sec | Includes 350 ms TXDELAY |
| Robust (rate 1/2) | 1200 | ~1.5 sec | Costas removes need for TXDELAY |
| Standard (rate 2/3) | 1600 | ~1.2 sec | Same modem, punctured |
| Fast (rate 3/4) | 1800 | ~1.0 sec | Same modem, more aggressive puncture |

For a 16-byte short message:

| Transport | Frame airtime |
|---|---:|
| AX.25 / AFSK 1200 | ~590 ms (TXDELAY-dominated) |
| Robust mode, 1 LDPC block | ~165 ms |

The single-block short-message win comes from the Costas-based
frame synchronisation removing the AFSK TXDELAY requirement;
this advantage is structural and will not erode under channel
noise.

## Out of scope

- Public APRS network interoperability (different protocol, no
  AX.25 wire format compatibility, no digipeater support).
- VOX-based PTT. The PTT path is BLE control of an IC-705. A
  classic USB-serial PTT cable on a PC is also supported. VOX is
  not a target — its pilot-tone preamble would balloon overhead
  on short frames.
- Mobile Doppler beyond ~10 Hz. The design targets pedestrian /
  stationary operation. Vehicular Doppler at VHF (8–13 Hz)
  remains characterisable but is not the primary operating point.
- VARA FM / Winlink interoperability.
- Adoption work, hardware product design, GUI polish.
- AX.25 / KISS / TNC framing layers. Caller decides upper layers.

## Phase plan

Each phase lands on `devel` as one or more atomic commits.

### Phase 0 — Revert old uvpacket and ship 0.3.0

Tracked separately (see 0.3.0 release tasks). 0.3.0 ships:

- LDPC unification under `LdpcParams` (already on `devel`)
- Q65 CRC-12 migration to `Q65Message::verify_info`
- Pipeline CRC reconstruction removal
- `DecodeResult.info: Box<[u8]>`
- `WsjtApCompatible` marker
- Multi-peak `coarse_sync` NMS for chained-signal support

The old uvpacket commits and the `PacketBytesMessage` codec
(added in 0.2.1 for the abandoned old uvpacket, no other users)
are reverted prior to tagging `v0.3.0`.

### Phase 1 — Modem core

New module `mfsk-core/src/uvpacket/`. The legacy module path is
reused; the protocol family inside is entirely new (different
sub-mode names, different FEC, different sync, different
framing).

- `protocol.rs` — three protocol ZSTs `UvRobust`, `UvStandard`,
  `UvFast` (all share the modem; differ only in FEC rate).
  Const constants per `Protocol` trait. All three pin
  `BITS_PER_SYMBOL = 2`, `NTONES = 4`, `BAUD = 1200`,
  `BT = 0.5`, `H = 0.5`, `TONE_SPACING_HZ = 600`.
  Audio centre 1700 Hz captured at synth time.
- `fec.rs` — new `Ldpc256_128Params` implementing `LdpcParams`.
  H matrix from CCSDS 231.1-O-1 (short LDPC). Puncturing patterns
  for rate 2/3 and 3/4 derived from the rate-1/2 mother code.
- `framing.rs` — frame header (mode 2b + block count 4b +
  app type 4b + sequence 6b + CRC-16 16b = 4 byte header),
  block packing, CRC-16 (CCITT-FALSE). Pure byte-pipe API; no
  `MessageCodec` impl. The header CRC-16 covers the full frame
  payload + header so a single CRC catches both header and
  payload corruption.
- `interleaver.rs` — block-interleaver over up to 16 × 256-bit
  blocks. Permutation seeded deterministically per frame.
- `sync.rs` — Costas 7×7 placement. Reuse FT4/FT8 Costas
  generator. Insert at frame head, every 4 blocks mid-frame, and
  at frame tail.
- `tx.rs` — bytes → header + frame → LDPC encode each block →
  puncture per mode → interleave → Costas-insertion → 4-GFSK
  synth. Reuses `core::dsp::gfsk::synth_f32` and
  `core::tx::codeword_to_itone`.
- `rx.rs` — Costas correlation → frequency / timing acquisition
  → mid-frame Costas tracking for AGC → 4-tone energy detection
  → de-interleave → de-puncture → LDPC decode → CRC-16 verify
  → frame reassembly.

Wiring: `mfsk-core/src/lib.rs` (`pub mod uvpacket;` under
`feature = "uvpacket"`); `registry.rs` (three `protocol_meta!`
lines); `tests/protocol_invariants.rs` (three invariant
assertions).

### Phase 2 — Characterisation harness

This is the *primary research deliverable*. Reproducible
PER-vs-SNR-vs-Doppler curves, machine-readable.

- `tests/uvpacket_awgn.rs` — `#[ignore]` SNR sweep per mode,
  -10 dB to +20 dB in 1 dB steps, 1000 frames per point. CSV
  output to `target/uvpacket_awgn_<mode>.csv`.
- `tests/uvpacket_rayleigh.rs` — `#[ignore]` (SNR, Doppler)
  matrix per mode. SNR -5 to +15 dB, Doppler {0.5, 1, 2, 5, 10,
  20} Hz. CSV output.
- `tests/uvpacket_diagnostics.rs` — exercises per-decode
  telemetry: Costas correlation peaks, LDPC iteration counts,
  per-symbol soft-decision distributions. Writes JSON
  diagnostic dumps that can be fed into matplotlib.

The Rayleigh channel model lives in
`mfsk-core/tests/common/channel.rs` (new). Sum-of-sinusoids or
filtered-Gaussian model with seedable RNG for reproducibility.

### Phase 3 — A/B comparison framework

Establishes honest comparison context. Without this, "robust"
claims have no reference point.

- `mfsk-core/tests/compare_afsk_baseline.rs` — pure-Rust AFSK
  1200 / Bell 202 demodulator (~300 lines, lift the bare
  algorithm — Direwolf is the gold standard but its source is
  too coupled to its full TNC stack to vendor in cleanly).
  Sweep same SNR/Doppler grid as Phase 2.
- `mfsk-core/tests/compare_ft4_baseline.rs` — FT4 demod is
  already in mfsk-core; just configure a comparable test
  scenario.
- `tools/plot_per_curves.py` (optional) — read CSVs from
  Phase 2 + Phase 3, emit overlay plots.

The comparison should establish, on the same audio recordings:

- AFSK 1200 SNR threshold for ≤ 1 % PER (AWGN)
- FT4 SNR threshold for ≤ 1 % PER (AWGN, mode-appropriate frame)
- uvpacket per-mode SNR thresholds for ≤ 1 % PER
- All three under matched 1 Hz Rayleigh

Expected outcome: uvpacket Robust beats AFSK by ~10 dB threshold
under fading; trails FT4 by a few dB but with much higher
throughput (FT4 is ~50 bps net, uvpacket Robust is 1200).

### Phase 4 — IC-705 BLE PTT integration

Field-test path. The decoder runs on a smartphone or laptop
browser; PTT is delivered over BLE to the IC-705.

- Reference webft8's `ble-transport.js` (CI-V over Web
  Bluetooth) and `cat.js` (Web Serial fallback). Patterns are
  already established; the integration here is mostly
  copy-and-adapt.
- Fixed `T_ptt_settle` (50–100 ms, calibrate empirically) between
  the BLE PTT-on command and audio start. This delay is *not*
  TXDELAY in the AFSK sense (audio modem settle); it is RF
  exciter settle. Costas synchronisation begins immediately when
  audio starts.
- WASM bindings for the modem in `uvpacket-web/` (cdylib) if a
  browser host is targeted. Leverage webft8's
  AudioContext/AudioWorklet pattern.

This phase is *optional* for 0.3.1. It can land as a tracked
add-on after the modem core (Phase 1) and characterisation
(Phase 2) have shipped.

### Phase 5 — Signed-QSL application

Resurrect `web-uvqsl/uvqsl-core` (the sibling repo, last in a
paused state) with the new modem. Most of the application code
is unchanged: signature scheme stays Bitcoin/Monacoin "Sign
Message", JSON wire format with pico_tnc field-name match.

Only the modem layer is replaced. The chain decoder in the old
uvqsl-core (which depended on `mfsk_core::uvpacket::ChainEncoder`)
is replaced by the new uvpacket framing — frames carry up to 256
bytes, comfortably more than the 214-byte signed-QSL payload, so
no chain layer is needed at all.

This collapses an entire layer of complexity from the old design.

### Phase 6 — APRS-style position reporting (private group)

Out of immediate 0.3.1 scope but planned. Short payloads (10–20
bytes for compressed lat/lon + altitude + status) fit in 1 LDPC
block, ~165 ms airtime, well-suited to a beacon every few
minutes within a private simplex group.

Defers to 0.3.2 or later.

## Verification

After each phase a defined set of tests must pass green; below is
the per-phase acceptance bar.

### Phase 1
- `cargo test -p mfsk-core --features uvpacket --release` green
- `cargo clippy --workspace --all-features --all-targets -- -D warnings` green
- Roundtrip tests per mode (encode → synth → decode → byte-equal)
- Multi-block frame roundtrip (16 LDPC blocks, 256 byte payload)
- Multi-signal coexistence test (two frames at distinct audio
  centres in same buffer; both decode)

### Phase 2
- AWGN PER curves match published CCSDS LDPC threshold ±0.5 dB
  for rate 1/2 (sanity check that the H matrix is correct)
- Rayleigh curves show graceful degradation, no PER cliffs from
  decoder pathologies
- CSV output is checked into the repo as committed reference data

### Phase 3
- AFSK baseline produces published AFSK / Bell 202 thresholds
  (≈ +10 dB SNR for 1% BER on AWGN)
- FT4 baseline matches WSJT-X published numbers within a dB
- Comparison plots produced and committed under
  `docs/uvpacket_per/`

### Phase 4
- BLE PTT round trip on IC-705 hardware (manual test, document
  the procedure)
- Audio path latency measured and documented

### Phase 5
- Signed-QSL roundtrip green in the sibling repo
- 214-byte QSL fits in a single uvpacket frame

## Open questions

- **Rate puncturing pattern**: standard CCSDS rate-1/2 puncturing
  patterns to reach 2/3 and 3/4? Or design specific patterns for
  fading? Defer to Phase 1 implementation; benchmark both during
  Phase 2.
- **Mid-frame Costas symbol budget**: 7 symbols × 3 placements =
  21 symbols overhead. Can be reduced to 5×3 = 15 if a Costas-5
  works at our SNR — needs a quick pilot study during Phase 1.
- **CRC polynomial**: CCITT-FALSE is the default pick, but
  CRC-16/T10-DIF or CRC-16/USB might be better-suited to the
  framing pattern. Negligible difference; pick one and stick.
- **AFSK baseline implementation**: write from scratch (~300 LOC,
  honest baseline) or vendor parts of an existing open-source
  decoder? Decide at Phase 3 start.
